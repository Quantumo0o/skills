# 交互式下单工作流程

## 角色说明

- **销售人员**：使用本技能的同事，一句话交代任务
- **客户**：销售人员的客户/供应商

## 核心原则

### OpenClaw工作原则

1. **提示完整清晰**：每步提示和引导信息一定要非常完整和清晰，只有清晰和完整的提示才能提高工作效率
2. **用户输入精简**：需要用户输入的流程应该尽可能精简
3. **领导交代任务模式**：模仿领导交代任务场景，一句话就把任务必要信息交代清楚，剩下就是OpenClaw自动完成，但是展示的成果应该非常完整而精确

### 示例

```
销售人员："帮牛建建下5吨Y163U1305276020000，新洋丰中磷，荆门东宝区泉口街道"

OpenClaw：（自动验证并展示完整结果）
         "订单确认：
         ━━━━━━━━━━━━━━━━━━━━━━━━━━━━
         客户：牛建建 (J620522007) ✓
         产品：Y163U1305276020000 - 洋丰复合肥 45% 15-15-15 ✓
         数量：5吨
         发货基地：新洋丰中磷 ✓
         收货地址：湖北省荆门市东宝区泉口街道馨梦缘公寓
         ━━━━━━━━━━━━━━━━━━━━━━━━━━━━
         
         确认下单？（确认/取消）"
```

## 输入模式

### 模式1：完整信息下单

销售人员一句话包含所有必要信息：

```
"帮[客户]下[数量][产品]，[发货基地]，[收货地址]"
```

**示例**：
```
"帮牛建建下5吨Y163U1305276020000，新洋丰中磷，荆门东宝区泉口街道"
```

**OpenClaw处理流程**：
1. 解析：客户=牛建建，产品=Y163U1305276020000，数量=5吨，基地=新洋丰中磷，地址=荆门...
2. 自动验证：
   - `getdealercode` 确认客户存在
   - `getproductlist` 验证产品可用性
   - `getdeliverybase` 确认基地有效
   - `getdealeraddresses` 匹配地址
3. 汇总确认 → 生成订单

### 模式2：部分信息下单

销售人员提供部分信息，OpenClaw补充缺失：

```
"帮牛建建下5吨Y163U1305276020000"
```

**OpenClaw处理**：
1. 解析：客户=牛建建，产品=Y163U1305276020000，数量=5吨
2. 自动验证客户和产品
3. 自动查询可用基地和地址
4. 缺失信息一次性询问：
   ```
   "好的，请选择：
   1. 发货基地：新洋丰中磷 / 武汉配送中心 / 成都仓库
   2. 收货地址：使用默认地址 / 输入新地址"
   ```

### 模式3：搜索式下单

销售人员需要浏览产品：

```
"看看牛建建能买什么复合肥"
```

**OpenClaw处理**：
1. 解析：客户=牛建建，关键词=复合肥
2. `getproductlist` 搜索该客户可购产品
3. 展示前5个结果：
   ```
   "牛建建可购买的复合肥：
   1. Y163U1305276020000 - 洋丰复合肥 45% 15-15-15
   2. Y163U1305276020001 - 洋丰复合肥 45% 16-16-16
   ...
   请选择并告诉我数量。"
   ```

### 模式4：订货单下单

销售人员发送订货单图片：

```
（发送订货单图片）
```

**OpenClaw处理**：
1. 解析图片，提取客户、产品、数量等信息
2. 自动验证客户和产品
3. 汇总确认：
   ```
   "识别结果：
   - 客户：牛建建 (J620522007) ✓
   - 产品：Y163U1305276020000 ✓
   - 数量：5吨
   - 基地：新洋丰中磷
   - 地址：荆门市东宝区泉口街道
   
   确认下单吗？"
   ```

## 自动验证流程

### 验证顺序

```
销售人员输入
    ↓
解析关键信息
    ↓
┌─────────────────────────────────┐
│ 并行验证                         │
│ ├─ 客户存在性 (getdealercode)    │
│ ├─ 产品可用性 (getproductlist)   │
│ ├─ 基地有效性 (getdeliverybase)  │
│ └─ 地址匹配 (getdealeraddresses) │
└─────────────────────────────────┘
    ↓
全部通过 → 汇总确认 → 下单
    ↓
有问题 → 一次性告知 + 提供选项
```

### 验证失败处理

**客户不存在**：
```
"客户'牛建建'不在您的客户列表中，请确认姓名或编码。
您的客户：张三(A123)、李四(B456)..."
```

**产品不可用**：
```
"牛建建无法购买Y163U1305276020000。
可选方案：
1. 换客户：张三、李四可购买此产品
2. 换产品：该客户可购买的类似产品有..."
```

**基地不匹配**：
```
"新洋丰中磷不支持配送到此产品，可选基地：武汉配送中心、成都仓库"
```

## 命令速查

```bash
# 获取销售员客户列表
xyfcli shop getdealercode "销售员编码"

# 验证产品可用性 / 搜索产品
xyfcli shop getproductlist -dealercode "客户编码" -search "产品编码或关键词"

# 获取发货基地
xyfcli shop getdeliverybase -productcode "产品编码" -dealercode "客户编码"

# 获取客户地址
xyfcli shop getdealeraddresses "客户编码"

# 下单
xyfcli order place -dealer "客户编码" -name "客户名称" -sales "销售编码" \
  -products "产品编码" -q "数量" -base "发货基地" -dest "收货地址"
```

## 最佳实践

### 销售人员输入建议

**推荐格式**：
```
"帮[客户]下[数量][产品]，[基地]，[地址]"
```

**完整示例**：
```
"帮牛建建下5吨Y163U1305276020000，新洋丰中磷，荆门东宝区泉口街道"
```

**信息越完整，交互越少**

### OpenClaw处理原则

1. **提示完整清晰**：每步提示和引导信息一定要非常完整和清晰
2. **自动优先**：能自动查询验证的，不追问
3. **并行处理**：多个验证同时进行
4. **一次询问**：缺失信息一次性收集
5. **清晰反馈**：验证结果用 ✓/✗ 标注，展示完整精确的结果
6. **提供选项**：出错时给出可选方案
