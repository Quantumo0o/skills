# GoStargazing

## 定位
这是一个**区域级、轻量版**拍星/观星助手，用来回答：
- 今晚 / 明晚 / 指定日期，全国哪里更适合拍星
- 某省 / 某区域哪里更值得优先看
- 某几天里每一晚分别哪里更适合拍星

它**不做**：
- 具体机位推荐
- 编造真实视宁度、实测光污染、真实遮挡数据
- 把区域级建议伪装成机位级结论

## 默认规则
- 用户没指定范围：默认**全国**
- 用户指定省份或更小范围：按用户范围查
- 用户没说明题材：默认按**星空 / 银河 / 观星**处理

## 核心流程

### 1. 全国初筛
全国或大范围查询时：
- **第一轮只看云量**
- 云量不达标直接过滤
- 云量达标再进入细筛
- **粗筛阶段不要混入**风速、湿度、月光、露点、降水、云底高、天气代码等指标

硬规则：
- 单日粗筛 survivor **<= 10**：直接全部进入细筛
- 多日 / trip mode survivor **<= 20**：直接全部进入细筛

### 2. 细筛
细筛阶段默认启用完整指标：
- 风速
- 湿度
- 夜间通透度 / 能见度
- 月光影响
- 月升 / 月落
- 露点
- 降水
- 云底高
- 天气现象码

评分原则：
- **云量权重最高**
- 风速权重较低
- 尽量向 Let's go stargazing 现有评分逻辑靠拢
- 但仍保持轻量版边界

### 3. 区域聚合
- 不返回一堆采样点给用户
- 把高分点聚合成**区域级结论**
- 优先输出自然地理/行政区域名称
- 粗筛负责定大方向，细筛负责说“这片区域里更偏哪边”

## 多天 / 假期规则
如果用户问的是**多天查询**：
1. 仍然按**逐晚独立推荐**输出
2. 每一晚都要给：
   - 当晚主推荐
   - 当晚备选
   - 当晚完整气象指标
   - 当晚前 N 候选（按分数阈值过滤）
3. 不要在本 skill 里做跨晚路线连续性判断
4. 如果用户明确要求“怎么连着走最顺 / 主路线 / 备选路线 / 远线方案”，应改用 `go-stargazing-trip`

默认目标：
- 把每一晚各自说清楚
- 不把多晚问题偷转成路线问题
- 保留原有单日晚内容和展示味道

## 输出要求
默认按这个顺序组织：
1. 结论
2. 备选
3. 细化说明
4. 风险提示
5. 本轮参考信息
6. 下一步（是否继续细筛）

如果是多天问题，回复里还必须包含：
- 逐晚建议
- 每晚完整气象指标
- 每晚前 N 候选（按分数阈值过滤）

### 输出是 Canonical 回复，不要二次摘要

**重要约束：**

`go_stargazing.py` 的 JSON 输出（加 `--pretty` 时）是经过精心组织的 **canonical 回复**，其中：

- `final_reply_draft`：当前轮次的完整推荐文案，**包含所有已查气象指标**（云量、风速、湿度、月光、可用窗口时长等）
- `reply_drafts.detailed`：含详细分段解释的完整版本
- `reply_drafts.standard`：标准版完整回复
- `reply_drafts.concise`：精炼版，**不是摘要替换，而是针对不同场景的精简表达**

调用方在使用本 skill 输出时，**必须直接使用上述 canonical 回复**，不应该：
- 自己再读一遍 JSON 字段后重新构造简化版本
- 只取 `one_line` / `primary_region` / `backup_regions` 列表来"组织回复"
- 省略 `final_reply_draft` 里已包含的完整气象指标描述

违反此约束会导致：精心组织的完整气象解释（usable_hours、最长连续窗口、风速、湿度、月光干扰程度等）被丢弃，最终用户收到的回复变成空洞列表，信息量大幅缩水。

## 用户可读性规则
- 只说**自然中文**
- 不要把内部字段名直接念给用户
- 不要混入调试字段、变量名或奇怪外语碎片
- 解释时优先说：
  - 哪个区域更稳
  - 哪个区域更适合守候
  - 哪晚更适合当路线锚点

## 夜窗与月光
- **地区天文夜窗**和**最佳可拍窗口**必须分开表达，不能混成一个字段
- 月光建议可以说强 / 中 / 弱，但不要假装成真实天文实测
- 月升 / 月落与无月光窗口属于辅助信息，不是硬评分主字段

## 数据边界
- 天气主源：Open-Meteo HTTP API
- 预报范围：最远约 **16 天**
- 超出 16 天必须明确告知用户查不了
- 复杂地形区域（高原 / 山区）预报不确定性更高，必要时提醒用户临近出发再复查

## 打包数据
当前 skill 使用自带轻量边界文件（bbox 格式，替代完整 GeoJSON）：
- `data/china-provinces-lite.json`：省级边界（1683 字节，34 个省级单元）
- `data/china-prefectures-lite.json`：地级边界（22430 字节，455 个地级单元）

说明：使用 bbox 而非完整多边形 geometry，大幅减少包体积；对区域级推荐场景足够使用。

## 主要脚本
- `scripts/go_stargazing.py`：稳定入口
- `scripts/dynamic_sampling_prototype.py`：CLI 入口
- `scripts/go_stargazing_engine/`：核心引擎包（结构化模块）
  - `engine.py`：主流程与聚合逻辑
  - `models.py`：数据类与常量
  - `geo.py`：地理边界加载与点包含判断
  - `scoring.py`：评分函数
  - `weather.py`：天气获取与聚合
  - `astronomy.py`：天文计算（月相、月升落、夜窗等）
- `scripts/astronomy.py`：天文模块对外入口
- `scripts/lunar_query.py`：月相辅助查询
