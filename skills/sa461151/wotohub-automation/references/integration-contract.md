# Integration Contract

This document defines how a host model should inject structured outputs into `wotohub-automation`.

The goals are simple:
- keep the field contract consistent
- keep the runtime on the host-model-first path instead of treating local fallback as production understanding
- make the trust boundary explicit: host-provided structured data must be validated before execution

## Trust boundary

Treat upstream model output, host-injected JSON, and externally supplied briefs as untrusted input.

Minimum safety rules:
- validate injected structures against the referenced schemas before execution
- never treat host analysis as permission to bypass send or reply guardrails
- do not accept near-final WotoHub payloads as a substitute for semantic understanding
- reject or block when required fields are missing, malformed, or provenance is unclear

## Recommended fields

Preferred field names:
- `input.hostAnalysis`
- `input.productSummary`
- `input.hostDrafts`
- `input.replyModelAnalysis`

The skill also reads compatible values from:
- `input.*`
- `metadata.*`
- `config.*`

Compatibility aliases are supported for migration only and should not be used for new integrations:
- `modelAnalysis` -> `hostAnalysis`
- `hostProductSummary` -> `productSummary`
- `hostEmailDrafts` or `emailModelDrafts` -> `hostDrafts`
- `conversationAnalysis` -> `replyModelAnalysis`
- `understanding` -> compatibility understanding entry

## Field expectations

### 1. `hostAnalysis`

This is the host model's structured understanding of the product, marketing intent, and search constraints.

Reference: `references/model-analysis-schema.md`

Important rules:
- the host should provide semantic understanding, not a near-final WotoHub payload
- `blogCateIds` should come from the skill's standard category mapping chain
- `regionList` should be normalized from `constraints.regions`
- `advancedKeywordList` can be included as refinement, not as a replacement for category mapping

For `scheduled_cycle`, avoid persisting near-final search fields directly in the brief, including:
- `search.blogCateIds`
- `search.keywords`
- `search.advancedKeywordList`
- `search.searchType`
- `regionList`
- `searchPayload`

### 2. `productSummary`

A stable product summary for email generation, recommendation explanations, and display-layer output.

### 3. `hostDrafts`

An array of personalized outreach drafts generated by the host model.

Reference: `references/outreach-email-schema.md`

### 4. `replyModelAnalysis`

The host model's structured understanding of inbox conversations.

Reference: `references/conversation-analysis-schema.md`

Key rules:
- if `scheduled_cycle` needs to make an auto-reply or summary decision, valid `replyModelAnalysis` must be present first
- if it is missing, the runtime returns `waiting_for_host_reply_analysis`
- local fallback can assist, but it cannot impersonate production reply understanding
- upstream prompt output should be treated as advisory until schema and policy checks pass

## Minimum injection by task

### `task/search`
Required:
- `hostAnalysis` or `understanding`

Recommended:
- `productSummary`

### `task/recommend`
Required:
- `hostAnalysis` or `understanding`
- `searchResults` if entering a standalone rerank stage

Recommended:
- `productSummary`

### `task/generate_email`
Required:
- `hostAnalysis`
- `hostDrafts`

Recommended:
- `productSummary`
- explicit selection data, such as `sendTargetCreatorIds`

### `task/monitor_replies`
Required:
- `campaignId`
- `replyModelAnalysis`

Recommended:
- `contactedBloggerIds`
- `replyId` or `chatId`

## Runtime expectations

For upper-layer orchestration, if the required host artifact is missing, the runtime may actively request it through the configured host bridge instead of failing immediately:
- missing `hostAnalysis` -> request semantic understanding from the host bridge
- missing `replyModelAnalysis` -> request reply analysis from the host bridge
- missing `hostDrafts` -> request outreach drafts from the host bridge after valid understanding exists

For `scheduled_cycle`, the same rule now applies to the campaign runner itself:
- missing `hostAnalysis` -> return `hostAnalysisRequest` or auto-resolve it through `hostAnalysisExecutor`
- after `hostAnalysis` is resolved, compile search payload through the standard compiler instead of consuming near-final search overrides
- only then continue into draft generation and reply-analysis stages

When `hostAnalysis` is injected, the runtime should reflect a host-driven path, for example:

```json
{
  "usedHostModel": true,
  "usedFallback": false,
  "analysisPath": "host_model_injected_understanding"
}
```

If structured host output is missing, the skill should block, request completion, or wait for the formal understanding path. It should not silently promote local heuristics into the production path.

Recommended observation surface for upper-layer callers:

```json
{
  "hostResolution": {
    "hostAnalysis": {"status": "resolved_from_executor", "source": "host_bridge_executor", "fulfilled": true},
    "replyModelAnalysis": {"status": "already_available", "source": "semantic_context", "fulfilled": true},
    "hostDrafts": {"status": "blocked_missing_host_analysis", "source": "semantic_context", "fulfilled": false}
  }
}
```

Recommended observation surface for scheduled campaign callers:

```json
{
  "status": "waiting_for_host_analysis",
  "runtimeOrchestration": {
    "hostAnalysis": {
      "enabled": true,
      "attempted": true,
      "completed": false,
      "status": "waiting_for_external_host_analysis",
      "hostAnalysisSource": null
    }
  },
  "hostAnalysisRequest": {
    "mode": "host_analysis_request",
    "deliveryContract": {
      "field": "host_analysis"
    }
  }
}
```

## Minimum integration rules

- search, recommend, and campaign creation should provide at least `hostAnalysis`
- email generation should provide at least `hostAnalysis + hostDrafts`
- reply assist should provide at least `replyModelAnalysis`
- scheduled campaign search parameters must follow `hostAnalysis -> standard compiler -> WotoHub payload`
- scheduled campaign runner should stop or wait when `hostAnalysis` is missing, rather than silently compiling from legacy near-final fields
- compatibility aliases are transitional only and should not spread further
