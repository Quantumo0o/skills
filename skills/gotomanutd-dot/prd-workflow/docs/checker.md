# PRD 质量检查项清单

**版本**: v1.0  
**生效日期**: 2026-04-05  
**说明**: 本文档定义了 PRD 生成过程中的质量检查项，分为核心检查项、完善检查项和优化检查项三类。

---

## 📋 检查项分类

| 类别 | 数量 | 说明 | 强制级别 |
|------|------|------|---------|
| **核心检查项 (CORE)** | 3 项 | 业务逻辑完整性，必须通过 | 🔴 强制 |
| **完善检查项 (COMPLETE)** | 5 项 | 需求描述完善度，建议通过 | 🟡 建议 |
| **优化检查项 (OPTIMIZE)** | 5 项 | 体验优化项，可选通过 | 🟢 可选 |

---

## 🔴 核心检查项 (CORE)

### CORE-1: 业务规则完整性

**检查点**:
- [ ] 所有业务场景是否都有对应的处理规则
- [ ] 规则之间是否存在冲突或矛盾
- [ ] 规则是否有明确的优先级定义
- [ ] 边界条件是否有明确定义

**检查问题**:
1. 是否存在未定义处理规则的业务场景？
2. 不同规则之间是否存在逻辑冲突？
3. 当多个规则同时适用时，优先级如何确定？
4. 极端情况（最大值、最小值、空值）是否有处理规则？

**验收标准**:
- 所有业务场景覆盖率 100%
- 规则冲突数 = 0
- 优先级定义清晰度 ≥ 90%

---

### CORE-2: 输入输出定义

**检查点**:
- [ ] 所有功能的输入参数是否有明确定义
- [ ] 所有功能的输出结果是否有明确定义
- [ ] 输入数据的格式、类型、范围是否有约束
- [ ] 输出数据的格式、类型、范围是否有约束

**检查问题**:
1. 每个功能模块的输入参数是否完整列出？
2. 每个功能模块的输出结果是否完整列出？
3. 输入数据的合法性校验规则是否明确？
4. 输出数据的格式规范是否明确？

**验收标准**:
- 输入参数定义完整率 100%
- 输出结果定义完整率 100%
- 数据约束明确率 ≥ 95%

---

### CORE-3: 异常处理

**检查点**:
- [ ] 所有可能的异常情况是否有识别
- [ ] 每种异常情况是否有对应的处理策略
- [ ] 异常处理是否有用户友好的提示
- [ ] 异常恢复机制是否有定义

**检查问题**:
1. 是否识别了所有可能的异常情况（网络、数据、权限、系统等）？
2. 每种异常情况是否有明确的处理流程？
3. 用户看到的错误提示是否清晰、可理解？
4. 系统是否支持异常后的恢复操作？

**验收标准**:
- 异常情况识别覆盖率 ≥ 90%
- 异常处理策略完整率 100%
- 用户提示友好度 ≥ 85%

---

## 🟡 完善检查项 (COMPLETE)

### COMPLETE-1: 用户角色定义

**检查点**:
- [ ] 所有用户角色是否有明确定义
- [ ] 每个角色的权限范围是否清晰
- [ ] 角色之间的关系是否有说明
- [ ] 角色切换机制是否有定义

**检查问题**:
1. 系统涉及哪些用户角色？
2. 每个角色可以执行哪些操作？
3. 不同角色之间的权限边界在哪里？
4. 用户是否可以切换角色？如何切换？

**验收标准**:
- 角色定义完整率 100%
- 权限描述清晰度 ≥ 90%

---

### COMPLETE-2: 业务流程覆盖

**检查点**:
- [ ] 主流程是否有完整描述
- [ ] 分支流程是否有完整描述
- [ ] 流程之间的衔接是否清晰
- [ ] 流程状态是否有明确定义

**检查问题**:
1. 核心业务流程是否有完整的步骤描述？
2. 条件分支（if/else）是否有明确说明？
3. 不同流程之间如何切换和衔接？
4. 每个流程节点的状态是否明确？

**验收标准**:
- 主流程覆盖率 100%
- 分支流程覆盖率 ≥ 85%
- 状态定义清晰度 ≥ 90%

---

### COMPLETE-3: 数据一致性

**检查点**:
- [ ] 数据来源是否有明确说明
- [ ] 数据存储方式是否有定义
- [ ] 数据同步机制是否有说明
- [ ] 数据冲突解决策略是否有定义

**检查问题**:
1. 数据从哪里来？（数据库、API、用户输入）
2. 数据如何存储？（表结构、字段定义）
3. 多系统/多端数据如何保持同步？
4. 数据冲突时以哪个为准？

**验收标准**:
- 数据来源明确率 100%
- 存储定义完整率 ≥ 90%
- 同步策略清晰度 ≥ 85%

---

### COMPLETE-4: 接口定义完整性

**检查点**:
- [ ] 所有外部接口是否有定义
- [ ] 接口请求参数是否有说明
- [ ] 接口响应格式是否有说明
- [ ] 接口错误码是否有定义

**检查问题**:
1. 系统需要调用哪些外部接口？
2. 每个接口的请求参数（名称、类型、必填）是否明确？
3. 每个接口的响应数据结构是否明确？
4. 接口错误码和错误信息是否有定义？

**验收标准**:
- 接口定义完整率 100%
- 参数说明清晰度 ≥ 95%

---

### COMPLETE-5: 验收标准可测性

**检查点**:
- [ ] 每个功能是否有明确的验收标准
- [ ] 验收标准是否可量化
- [ ] 验收标准是否可执行
- [ ] 验收标准是否有优先级

**检查问题**:
1. 每个功能点是否有对应的验收标准？
2. 验收标准是否可以用数字或明确的是/否来判断？
3. 测试人员是否可以根据验收标准执行测试？
4. 验收标准的优先级是否明确（P0/P1/P2）？

**验收标准**:
- 功能验收标准覆盖率 100%
- 可量化率 ≥ 90%
- 可执行率 100%

---

### COMPLETE-6: 验收标准格式（GWT）

**检查点**:
- [ ] 验收标准是否使用 GWT 格式（Given-When-Then）
- [ ] 每条验收标准是否包含三要素（前置条件/操作/预期结果）
- [ ] GWT 格式是否正确（Given 描述前提、When 描述操作、Then 描述结果）
- [ ] 验收标准是否使用中文描述

**检查问题**:
1. 验收标准是否采用"Given...When...Then..."格式？
2. 每条验收标准是否包含前置条件（Given）？
3. 每条验收标准是否包含操作/事件（When）？
4. 每条验收标准是否包含预期结果（Then）？
5. 验收标准是否使用中文而非英文？

**验收标准**:
- GWT 格式覆盖率 ≥ 80%
- 三要素完整率 100%
- 中文描述率 100%

---

## 🟢 优化检查项 (OPTIMIZE)

### OPTIMIZE-1: 性能指标定义

**检查点**:
- [ ] 关键操作是否有响应时间要求
- [ ] 系统并发能力是否有定义
- [ ] 数据处理量是否有预估
- [ ] 性能优化策略是否有说明

**检查问题**:
1. 关键操作的响应时间要求是多少？（如：页面加载 < 2s）
2. 系统需要支持多少并发用户？
3. 预计每天/每月处理多少数据量？
4. 有哪些性能优化策略？（缓存、异步、分页等）

**验收标准**:
- 关键性能指标定义率 ≥ 80%
- 优化策略可行性 ≥ 85%

---

### OPTIMIZE-2: 安全合规检查

**检查点**:
- [ ] 敏感数据是否有加密说明
- [ ] 权限验证是否有定义
- [ ] 合规要求是否有说明
- [ ] 审计日志是否有定义

**检查问题**:
1. 敏感数据（密码、身份证号等）如何加密存储？
2. 权限验证在哪些环节执行？
3. 是否符合行业合规要求？（如金融、医疗）
4. 关键操作是否有日志记录？

**验收标准**:
- 安全措施覆盖率 ≥ 85%
- 合规要求明确率 ≥ 90%

---

### OPTIMIZE-3: 可扩展性设计

**检查点**:
- [ ] 系统是否支持功能扩展
- [ ] 配置是否支持动态调整
- [ ] 架构是否支持水平扩展
- [ ] 未来演进方向是否有说明

**检查问题**:
1. 新增功能模块是否会影响现有功能？
2. 系统参数是否可以通过配置调整？
3. 系统是否支持增加服务器来提升性能？
4. 未来 1-3 年的演进方向是什么？

**验收标准**:
- 扩展性设计完整率 ≥ 75%
- 配置灵活度 ≥ 80%

---

### OPTIMIZE-4: 用户体验优化

**检查点**:
- [ ] 操作流程是否简洁
- [ ] 界面反馈是否及时
- [ ] 错误提示是否友好
- [ ] 帮助文档是否有说明

**检查问题**:
1. 用户完成核心任务需要多少步？
2. 操作后是否有明确的反馈？（成功/失败/加载中）
3. 错误提示是否告诉用户如何解决问题？
4. 是否有用户帮助文档或引导？

**验收标准**:
- 操作步骤简洁度 ≥ 80%
- 反馈及时率 ≥ 90%
- 提示友好度 ≥ 85%

---

### OPTIMIZE-5: 监控告警设计

**检查点**:
- [ ] 关键指标是否有监控
- [ ] 异常情况是否有告警
- [ ] 告警通知渠道是否有定义
- [ ] 告警级别是否有分级

**检查问题**:
1. 哪些关键指标需要监控？（CPU、内存、请求量等）
2. 什么情况下触发告警？
3. 告警通过什么渠道通知？（短信、邮件、钉钉）
4. 告警级别如何分级？（P0/P1/P2/P3）

**验收标准**:
- 关键指标监控率 ≥ 80%
- 告警策略完整率 ≥ 85%

## 📊 检查项使用说明

### 在 PRD 生成流程中的集成点

| 阶段 | 使用的检查项 | 集成方式 |
|------|-------------|---------|
| **需求分解** | CORE-1, CORE-2, CORE-3 | 生成检查清单，指导 AI 生成 |
| **PRD 生成** | COMPLETE-1 ~ COMPLETE-5 | 作为生成提示词的一部分 |
| **质量检查** | 全部 13 项 | 作为质量门禁的评分标准 |
| **优化建议** | OPTIMIZE-1 ~ OPTIMIZE-5 | 生成优化建议报告 |

### 检查项加载器 API

```javascript
const loader = require('./workflows/check_items_loader.js');

// 加载全部检查项
const allItems = await loader.load();

// 按阶段加载检查项
const coreItems = await loader.loadForStage('decomposition');
const completeItems = await loader.loadForStage('prd_generation');
const optimizeItems = await loader.loadForStage('quality_check');

// 生成阶段提示词
const prompt = await loader.generatePrompt('decomposition');
```

### 检查项评分规则

| 等级 | 得分 | 说明 |
|------|------|------|
| **完全符合** | 100% | 所有检查点均满足 |
| **基本符合** | 70-99% | 大部分检查点满足 |
| **部分符合** | 40-69% | 部分检查点满足 |
| **不符合** | <40% | 大部分检查点不满足 |

**通过标准**:
- 核心检查项：必须 100% 通过
- 完善检查项：≥80% 通过
- 优化检查项：≥60% 通过
